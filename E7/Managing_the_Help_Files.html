Managing the Help Files

E7 uses TortoiseSVN for managing files on their network. TortoiseSVN is a free open-source program for version control and is available at http://tortoisesvn.net/. In staff meetings, you will frequently hear the terms "checkout" and "check-in", and this refers to using TortoiseSVN.

Checking Out the Help Files

For the first time editing any help files, you will need to check them out from their network location using TortoiseSVN. By default this action will copy the files and folder structure to the My Documents folder on your PC. Preferred practice is to create and use a sub-folder to contain these help files.

  1. In Windows Explorer, create a sub-folder in your My Documents folder that will contain the help files you will edit (e.g., ...\E7).
  2. Select the folder you just created and right-click on it. A context menu appears and you should see SVN Checkout... about a 1/3 of the way down.
  3. Click on SVN Checkout... The Checkout dialog window opens.

  4. In the top field, "URL of repository:", type in or select from the drop-down where the help files are stored. Help files are stored at svn://red-repo.securityinspection.com/repos/HelpDocumentation. To checkout a specific Help directory, e.g. GODDARD, click on the ellipsis button that's to the right of the field and drill down as needed at the Tortoise Repository Browser window that appears, e. g., .../GODDARDHelp/trunk. All current released versions are stored in the trunk folder, and files for the next release are typically in the branch folder.
  5. Click the OK button after you have selected the folder/files you want to checkout. The files in the path you selected will download to your PC.
    Some notes on TortoiseSVN and file checkout:
    • A complete install of TortoiseSVN includes a very detailed help file. Highly recommended reading if you're new to TortoiseSVN.
    • If you checkout the entire HelpDocumentation folder, you will be copying over 80,000 files stored in over 20,000 folders totaling approximately 1.7GB of disk space (as of July 2011) to your PC.
    • The file/folder icons in the checkout directory will change slightly depending on their status. If the file/folder is unchanged, the icon will have a checkmark. When you make and save changes to an existing file, the file and folder icon will have an exclamation point. Finally, if you create a new file/folder, a question mark will appear in their icon (more about this later).
    • Be aware that these files are still available on the network for others to checkout and work on if necessary.
    • To compile a revised Help file, you must also checkout Support/Custom_OTs (not necessary if you checked out the entire HelpDocumentation folder).

About the Help File Folder Structure

For editing the next version of the help files, you most likely will need to go to the "...\APPNAMEHelp\Branches\VersionNumber" folder (where VersionNumber uses a "1.2.3" format). If there are multiple VersionNumber folders, use the highest number (unless you have been told otherwise).

In this folder you will find a number of XML files and a least a couple of files with an extension starting with "dita". These are the files you will be using in XMetaL. Most likely there will also be a couple of subfolders, "\Graphics" and "\webhelp_out". Add the "\Graphics" folder if it's not there as that is where you will store the graphics that you want to add to the help files. The "\webhelp_out" folder is automatically created and populated by XMetaL when it generates a WinHelp.

CAUTION:
At each folder level, there will also be a hidden folder "\.svn". This is used by TotoiseSVN for version control. Don't mess with this folder or its contents.

About the Help Files

As noted in the previous topic, there a number of XML files and a least a couple of files with an extension starting with "dita" in the "...\APPNAMEHelp\Branches\VersionNumber" folder. Combined, these files will be the source for the winhelp that the application will use. The help content development process uses the Darwin Information Typing Architecture (DITA). DITA is a standard XML data model for authoring and publishing.

In the table below describes the key topic types (per the DITA specification) that the E7 help uses, and also the filename conventions.

Content Type File Naming Description
Generic filename.xml

From DITA Architectural Specification (available via XMetaL Help > DITA Specifications): "The generic or unspecialized topic type provides the base for other specialized topic types, and also provides a place to author content that does not belong in existing specialized types."

XMetaL does not apply a prefix to XML files it creates that use this DITA topic type. Within this file, you cannot use other DITA topic types using XMetaL.

Concept c_filename.xml

From DITA Architectural Specification: "DITA concept topics answer ″What is...″ questions."

XMetaL applies a "c_" prefix to XML files it creates that use this DITA topic type. Within this file, you cannot use other DITA topic types using XMetaL.

Reference r_filename.xml

From DITA Architectural Specification: "Reference topics describe regular features of a subject or product, such as commands in a programming language."

XMetaL applies a "r_" prefix to XML files it creates that use this DITA topic type. Within this file, you cannot use other DITA topic types using XMetaL.

Task t_filename.xml

From DITA Architectural Specification: "Task topics answer ″How do I?″ questions, and have a well-defined structure that describes how to complete a procedure to accomplish a specific goal."

XMetaL applies a "t_" prefix to XML files it creates that use this DITA topic type. Within this file, you cannot use other DITA topic types using XMetaL.

Composite filename.xml Not part of the DITA specification, XMetaL allows you to create an XML file where you can use any or all of the official DITA topic types. As of this writing, the only help topic at E7 that uses this format is this topic that you're reading now.
Reuseable Component filename.xml or filenameDEF.xml

The actual DITA content type (and XMetaL file type) is generic, but the content is tagged as Reuseable Content. These files are inserted when the same term, sentence, paragraph, etc. is used in multiple topics, much like a Routine. When you edit a Reuseable Component, the revision is cascaded globally. So, for example, we change how we derive an Ident, you only need to revise the IdentDef.xml file for each product.

In a help topic, a Reuseable Component is wrapped with the <ph> Element tag with a light gray highlight. To open a component from a topic, right-click and select Open Referenced file. Occasionally, component files are compressed in View. To display the contents, right-click and select Refresh All References.

DITA Map filename.ditamap A DITA map defines how the topics are to be organized in the final output. The map is commonly used as a TOC, but unlike a word processor that generates a TOC from the topic headers, you place the topic headers in the map where you want that topic to appear.
DITA Values filename.ditaval

Checking In Your Changes

At regular intervals, typically end-of-day, you should upload or check-in your changes to the network. The primary reason is for backup but the other reason is that your colleagues will be able to use the files with your changes. Given that multiple persons can be editing the same file, uploading your changes may be a 3 step process: updating your local copy, resolving any conflicts, and committing your changes. A more detailed process is below:

  1. In Windows Explorer, right-click the folder that contains all the files you changed and/or added to the help and select SVN Update from the menu. A TortoiseSVN progress window appears displaying what files on your local copy are updated and how in the Action column. If the Action column states "Updated" or "Added", this is a file that someone else changed (but not you) or added. If the Action column states "Merged", a colleague has made changes to a file that you also edited, but these changes are in a different area of the file. If the Action column states "Conflicted" (it will be in red) this means that a colleague made changes to the same content as you in that file.
  2. Click OK to close the TortoiseSVN progress window.
  3. Resolve any Conflicted files. See Resolving Conflicts in the TortoiseSVN help on how to fix this.
  4. In Windows Explorer, right-click the folder that contains all the files you changed and/or added to the help and select SVN Commit from the menu. After scanning your PC and the network, a TortoiseSVN Commit window appears listing all the files in your folder path with a checkbox next to their name. TortoiseSVN will automatically mark this box for files that you checked out and modified. Unmodified and new files will not have a check mark.
  5. Check the box for the new files you want to upload.
  6. Click OK to begin the upload. A network login window appears.
  7. Enter the required network login information and click OK. The TortoiseSVN Commit progress window appears.
  8. Click the OK button to close the progress window after it's finished uploading.
